# AGENTS.md - AI Document Presentation V3

This file provides guidelines for agentic coding agents working in this repository.

## Agent Behavior (IMPORTANT)

- **Always plan and get user approval before making any code changes**
- Never make assumptions about user intent - ask clarifying questions
- Read existing code before making modifications
- Follow the code style guidelines below

## Project Overview

AI Document Presentation V3 transforms educational documents into animated video presentations with AI Avatar narration, Manim animations, and synchronized visual content. It's a Python Flask API with LLM-powered pipeline.

**Note:** V3 generates **Manim** animations, not Three.js. Three.js infrastructure exists but is not actively used in the pipeline.

## V3 Pipeline (Visual-First)

### Entry Point
- **File:** `core/pipeline_v3.py`
- **Function:** `run_v3_pipeline()`

### Pipeline Flow

| Phase | File | Purpose |
|-------|------|---------|
| 1. Chunker | `core/utils/smart_partitioner.py` → `SmartPartitioner` | Partitions markdown into chunks |
| 2. Director | `core/partition_director_generator.py` → `PartitionDirectorGenerator` | LLM generates sections from chunks |
| 3. Validation | `core/v3_validator.py` | Validates V3 output |
| 4. Avatar Gen | `core/agents/avatar_generator.py` | Generates AI avatar MP4s (master clock) |
| 5. Manim Code | `core/agents/manim_code_generator.py` | Generates Python code from `manim_scene_spec` |
| 6. Timing Enforcer | `core/manim_timing_enforcer.py` | Scales animations to match avatar duration |
| 7. Manim Render | `render/manim/manim_runner.py` | Executes Manim CLI → .mp4 |
| 8. Quiz Cards | `core/quiz_card_generator.py` | Generates quiz cards |
| 9. WAN/LTX Video | `core/renderer_executor.py` | Generates videos for `renderer:video` sections |

### Prompt Files

| File | Handles |
|------|---------|
| `core/prompts/director_v3_partiton_prompt.txt` | Content + Example sections + Understanding Quiz |
| `core/prompts/director_global_prompt.txt` | Intro + Summary + Memory + Recap + Document Quiz |

### Architecture Rules

#### Section-Level vs Beat-Level Renderer

| Level | content/example/quiz/recap | intro/summary/memory |
|-------|--------------------------|---------------------|
| **Section** | `"video"` | `"none"` |
| **Beat** (in `render_spec.segment_specs`) | manim / image_to_video / text_to_video / infographic / image | N/A |

#### Beat-Level Renderers (LLM decides per beat)
- **manim** - molecules, formulas, equations, geometry, graphs, STEM diagrams
- **image_to_video** - biology, anatomy, pharmacology (needs reference image)
- **text_to_video** - history, geography, narrative (mood only)
- **infographic** - static labelled diagrams, comparisons
- **image** - displays source document image directly

#### Display Directives
```json
// Content segments (manim/image_to_video/etc)
{ "text_layer": "hide", "visual_layer": "show", "avatar_layer": "show" }

// None segments (intro/summary/memory)
{ "text_layer": "hide", "visual_layer": "hide", "avatar_layer": "show" }
```

### Mandatory Fields

| Field | Where | Description |
|-------|-------|-------------|
| `renderer_reason` | Every `segment_spec` | One sentence explaining renderer choice |
| `manim_scene_spec` | renderer: "manim" | 100+ words, explicit timing, ends with "Duration exactly Xs." |
| `image_prompt` | renderer: image_to_video/infographic | 100+ words, all labels quoted |
| `video_prompt` | renderer: image_to_video/text_to_video | 80+ words, mood only, "15 seconds. 24fps." |
| `understanding_quiz` | Every content/example section | Mandatory quiz after section |
| `explanation_visual` | Every quiz question | Visual shown during explanation |
| `explanation_script` | Every quiz question | **4th narration script** - mandatory |

### Quiz System

- **4 Avatar Clips per question** (mandatory):
  1. `question` - reads question + options
  2. `correct` - positive reaction
  3. `wrong` - warm correction
  4. `explanation` - narrates over explanation_visual

- **Quiz is HTML/JS** - NOT Manim, player handles A/B/C/D buttons
- **Two Quiz Types:**
  - Understanding Quiz - generated by partition prompt, 1 per content/example section
  - Document Quiz - generated by global prompt, from source document MCQs

### Known Bug
- `manim_code_generator.py` writes Python code back into `presentation.json` 
- Should only store in `manim/` folder, not in the JSON

## Code Style Guidelines

### General Principles
- Write clean, readable code with clear intent
- Keep functions focused and single-purpose
- Use descriptive names for variables, functions, and classes

### Type Hints
- **Always use type hints** for function parameters and return types
- Use built-in types: `str`, `int`, `float`, `bool`, `list`, `dict`
- For complex types, use `typing` module: `Optional`, `Dict`, `List`, `Tuple`, `Any`
- Example:
  ```python
  def process_job(job_id: str, options: Optional[Dict[str, Any]] = None) -> Dict[str, Any]:
  ```

### Naming Conventions
- **Variables/functions**: snake_case (e.g., `job_manager`, `load_jobs_index`)
- **Classes**: PascalCase (e.g., `BaseAgent`, `PipelineError`)
- **Constants**: UPPER_SNAKE_CASE (e.g., `JOBS_DIR`, `DEFAULT_MODEL`)
- **Private methods**: prefix with underscore (e.g., `_build_semantic_retry_guidance`)

### Imports
- Group imports in this order:
  1. Standard library
  2. Third-party packages
  3. Local application imports
- Sort alphabetically within each group
- Example:
  ```python
  import json
  import os
  from pathlib import Path
  from typing import Dict, List, Optional, Tuple

  import jsonschema
  import tenacity
  from openai import OpenAI
  from flask import Flask, request, jsonify

  from core.pipeline import process_pdf_to_videos
  from core.agents.base_agent import BaseAgent
  ```

### Docstrings
- Use Google-style or simple docstrings for all public functions
- Include description, args, and return type
- Example:
  ```python
  def load_jobs_index() -> Dict[str, dict]:
      """Load jobs index from disk."""
      ...
  ```

### Error Handling
- Use custom exception classes for domain errors
- Example from codebase:
  ```python
  class AgentError(Exception):
      """Raised when an agent fails after retries."""
      def __init__(self, agent_name: str, message: str, errors: List[str], attempts: int):
          self.agent_name = agent_name
          self.message = message
          self.errors = errors
          self.attempts = attempts
          super().__init__(f"[{agent_name}] {message} after {attempts} attempts: {errors}")
  ```
- Use specific exception types (`FileNotFoundError`, `JSONDecodeError`, etc.)
- Log errors appropriately with context

### Logging
- Use the custom `log()` function for real-time console output:
  ```python
  def log(msg: str):
      """Print with immediate flush for real-time logging."""
      print(msg)
      sys.stdout.flush()
  ```
- Use Python's `logging` module for structured logging in API

### Thread Safety
- Use locks for shared state modifications:
  ```python
  from core.locks import presentation_lock, analytics_lock
  with presentation_lock:
      # safe to modify presentation.json
  ```

### JSON Handling
- Use `json` module for JSON operations
- Use `jsonschema` for validation
- The codebase has utilities for fixing malformed LLM JSON:
  ```python
  from core.llm_client import fix_malformed_json, was_truncated
  ```

### LLM Integration
- Use OpenAI client via OpenRouter for LLM calls
- Default model: `google/gemini-2.5-flash`
- Strong model: `google/gemini-2.5-pro`
- Use tenacity for retries:
  ```python
  from tenacity import retry, stop_after_attempt, wait_exponential

  @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
  def call_llm(...):
      ...
  ```

### Agent Architecture
Follow the `BaseAgent` pattern for LLM-powered agents:
1. Define input contract (type hints)
2. Load system and user prompts from files
3. Call LLM with prompts
4. Validate output against JSON schema
5. Return validated output or raise `AgentError`

Example:
```python
class MyAgent(BaseAgent):
    name = "MyAgent"
    system_prompt_file = "my_agent_system.txt"
    user_prompt_file = "my_agent_user.txt"
    output_schema_file = "my_agent_schema.json"
    model = DEFAULT_MODEL
    temperature = 0.3
```

### Pydantic Models
- Use Pydantic for data validation and serialization
- Define models for API requests/responses

### File Paths
- Use `pathlib.Path` for file path operations
- Define base paths as constants:
  ```python
  JOBS_DIR = Path("player/jobs")
  JOBS_INDEX_FILE = JOBS_DIR / "jobs_index.json"
  ```

### Configuration
- Use environment variables via `python-dotenv`
- Load in this order:
  ```python
  from dotenv import load_dotenv
  load_dotenv(override=True)
  ```

## Common Tasks

1. Create `core/pipeline_vN.py`

### Adding a New Agent
1. Create agent class inheriting from `BaseAgent`
2. Define prompts in `core/prompts/`
3. Define JSON schema in `schemas/v1.5/`

## Issue Tracking Process

When a bug or issue is identified, follow this process:

### 1. Investigation Phase
1. **Read existing documentation:**
   - Check `V3_SYNC_ISSUES.md` for similar issues
   - Check `implementation_plan.md` for related plans
   - Review relevant code files

2. **Debug the issue:**
   - Identify the specific job ID that has the issue
   - Examine `presentation.json` to understand data structure
   - Check Docker logs for errors
   - Verify file generation in job folders

3. **Document findings:**
   - Update `V3_SYNC_ISSUES.md` with root cause analysis
   - Include code examples, file paths, and error messages

### 2. Planning Phase
1. **Create implementation plan:**
   - Document the current behavior vs expected behavior
   - Show concrete examples using actual job IDs
   - Outline the solution with pseudo-code or flow diagrams

2. **Review with user:**
   - Present the plan in `implementation_plan.md`
   - Get user approval before implementing

### 3. Implementation Phase
1. **Implement only after user approval**
2. **Follow code style guidelines**
3. **Test the implementation**
4. **Update documentation**

### 4. Documentation Files

| File | Purpose |
|------|---------|
| `V3_SYNC_ISSUES.md` | Master issue tracker with status, root cause, and resolutions |
| `implementation_plan.md` | Detailed implementation plans for review and approval |

### Example: Adding a New Issue

```markdown
## ISSUE-ID 🔴 IN PROGRESS

- Explain what went wrong

**Example from Job `{job_id}`:**
- Show actual data structure

- Document the solution
```

## V3 Asset Generation Flow

When implementing asset generation for V3 sections:

```
1. READ presentation.json
   └── Find sections with renderer="video"
   └── Read render_spec.segment_specs

2. ROUTE by renderer type:
   ├── text_to_video     → LocalGPU/LTX video
   ├── image_to_video    → Gemini image + LocalGPU/LTX video
   ├── infographic       → Gemini image (static)
   └── manim             → V3 Manim pipeline

3. BATCH generation:
   ├── Phase 1: Gemini images (PARALLEL)
   ├── Phase 2: LTX videos (BATCHED - 3 concurrent)
   ├── Phase 3: Update presentation.json
   └── Phase 4: Manim (PARALLEL with Phase 1)

4. SAVE assets:
   ├── jobs/{id}/images/     → Gemini images
   ├── jobs/{id}/videos/    → LTX videos
   └── jobs/{id}/manim/      → Manim code + renders
```
