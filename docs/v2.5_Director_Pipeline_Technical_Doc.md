## 1. Architecture Overview (The "Director-Pointer" Model)

The V2.5 pipeline uses a **Parallel Director** architecture via the `PartitionDirectorGenerator` class. Unlike previous versions that generated content sequentially, V2.5 splits the work into specialized "Passes" using a "Partition & Conquer" strategy.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     V2.5 PARALLEL PIPELINE WORKFLOW                         │
└─────────────────────────────────────────────────────────────────────────────┘

                                   ┌──────────────┐
                                   │ Raw Markdown │
                                   └──────┬───────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│   PASS 0: SMART CHUNKER (Structure Analysis)                                │
│   [core/smart_chunker.py] + [core/utils/smart_partitioner.py]               │
│                                                                             │
│   1. Detect Block Types: Tables, Images, Formulas, Q&A.                     │
│   2. Extract Logic Topics: "Logical" chunks based on headers/context.       │
│   3. Output: Clean Markdown Chunks + Density Analysis.                      │
└─────────────────────────────────────────────────────────────────────────────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│   PASS 1: THE PARTITIONER (Concurrency Setup)                               │
│   [core/partition_director_generator.py]                                    │
│                                                                             │
│   Class: `PartitionDirectorGenerator`                                       │
│   Method: `generate_presentation_partitioned()`                             │
│                                                                             │
│   1. Splits document into N physical chunks.                                │
│   2. Spawns ThreadPoolExecutor (Max 12 workers).                            │
│   3. Submits 1 Global Worker + N Content Workers simultaneously.            │
└─────────────────────────────────────────────────────────────────────────────┘
                                          │
      ┌───────────────────────────────────┼───────────────────────────────────┐
      │                                   │                                   │
      ▼                                   ▼                                   ▼
┌─────────────┐                     ┌─────────────┐                     ┌─────────────┐
│  WORKER 1   │                     │  WORKER 2   │                     │  GLOBAL     │
│ (Topic 1)   │                     │ (Topic 2)   │                     │  WORKER     │
├─────────────┤                     ├─────────────┤                     ├─────────────┤
│ LLM: Direct │                     │ LLM: Direct │                     │ LLM: Direct │
│ Specific    │                     │ Specific    │                     │ Intro/Summ/ │
│ Chunk       │                     │ Chunk       │                     │ Recap/Mem   │
└──────┬──────┘                     └──────┬──────┘                     └──────┬──────┘
      │                                   │                                   │
      └───────────────────────────────────┼───────────────────────────────────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│   PASS 3: THE MERGER & IMAGE INJECTION                                      │
│   [core/partition_director_generator.py]                                    │
│                                                                             │
│   1. Merge all JSON sections (Global + Content Results).                    │
│   2. Sort Sections (Intro -> Summary -> Content -> ... -> Recap).           │
│   3. **Pipeline Fix**: `inject_missing_image_ids` scans for null            │
│      visual references and maps them to markdown images.                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Component Details

### 2.1 Pass 0: Smart Chunker
**File**: `core/smart_chunker.py`
**Goal**: Structure preservation and intelligent segmentation.

*   **Block Detection**: Identifies specialized content:
    *   **Tables**: Preserved as `| table | structure |`.
    *   **Images**: Extracts `![alt](url)` to ensure they become **Visual Beats**.
    *   **Q&A**: Detects "Question:" pattern to recommend **Quiz** sections.

### 2.2 Pass 1 & 2: The Partition Director
**File**: `core/partition_director_generator.py`
**Class**: `PartitionDirectorGenerator`

The **Partitioner** physically divides the task. It does NOT process sequentially.
*   **Global Worker**: Reads the *entire* document to generate Context sections (Intro, Summary, Memory, Recap).
    *   Prompt: `director_global_prompt.txt`
*   **Content Workers**: Each worker receives ONE chunk + Full Context (Read Only).
    *   Prompt: `director_partition_prompt.txt`
    *   **Sync Splitter**: `_apply_sync_splitter` automatically breaks long narration (>40 words) into 15-second visual beats (`_beat_1`, `_beat_2`) to keep the video dynamic.

### 2.3 Pass 3: Manim Code Bridging
**File**: `core/pipeline_unified.py` (Phase 3)
**Executor**: `core/agents/manim_code_generator.py`
**Concurrency**: `ThreadPoolExecutor` (Max 5 workers).

*   **Logic**: The Director outputs a text `manim_scene_spec`.
*   **Bridge**: `ManimCodeGenerator` class reads this spec and writes actual Python code (`section_X.py`).
*   **Integration**: The code is injected back into the `presentation.json` before rendering.

### 2.4 Pass 5: Render Execution (Split-Phase)
**File**: `core/pipeline_unified.py`

*   **Manim (Math/Physics)**:
    *   **Blocking**: The pipeline WAITS (`render_all_topics` with `renderer_filter="manim"`).
    *   **Reason**: Math requires exact correctness. If code fails, we must retry immediately.

*   **WAN (Biology/History/Recap)**:
    *   **Async (Fire-and-Forget)**: Handled by a background thread (`submit_wan_background_job`).
    *   **Batching**: Processed in groups of 5 to avoid API rate limits.
    *   **Updates**: Real-time writing to `presentation.json` as videos finish.

### 2.5 Pass 6: Multi-Language Avatar Support (Async)
**File**: `core/agents/avatar_generator.py`
**API**: `POST /job/<job_id>/generate_avatar`

V2.5 allows generating avatars in multiple languages concurrently using the same narration text but dedicated regional speakers.

*   **Logic**: 
    1.  The system accepts a list of `languages` and a `speaker`.
    2.  It iterates through each language, selecting a regional speaker (if not provided).
    3.  It submits parallel jobs to the Kie.ai API for each section.
*   **Data Structure**: 
    *   `presentation.json` is updated with an `avatar_languages` object.
    *   Structure: `"avatar_languages": { "{lang}": { "speaker": "...", "status": "..." } }`.
*   **Storage**: 
    *   Avatars are stored in `avatars/{lang}/section_{id}_avatar.mp4`.
    *   Metadata is preserved in `avatars/metadata.json` for persistent tracking.
*   **Sanity Check Integration**: The `sanity_check.html` tool is updated to discover and validate these subfolders automatically.

## 3. Strict Validation Logic (V2.5 Bible)
**File**: `core/validators/v25_validator.py`

The pipeline enforces the "Director Bible" rules programmatically:

| Section | strict Rule | Validator Action |
| :--- | :--- | :--- |
| **Intro** | Clean Start | Rejects if `renderer != "none"` or if any Text/Visual layer is shown. |
| **Summary** | Bullet Points | Rejects if `visual_type != "bullet_list"`. |
| **Memory** | 5 Flashcards | Rejects if `count != 5` or narration segments != 6 (1 Intro + 5 Cards). |
| **Recap** | Cinematic | Rejects if `renderer != "video"` or `video_prompts < 5`. Enforces 80+ words per prompt. |
| **Quiz** | 3-Step Dance | Rejects if segments are not multiples of 3. Checks for "Introduce" -> "Pause" -> "Explain" flow. |

## 5. Troubleshooting & common Issues

### Q: Why is my Table not showing?
*   **Check**: Does the markdown table use standard `| header |` syntax?
*   **Fix**: Smart Chunker requires correct pipe syntax. It will be converted to a `visual_beat` of type `diagram`.

### Q: Why are visuals missing in Content?
*   **Check**: Did you provide images in the source markdown?
*   **Logic**: The `inject_missing_image_ids` function attempts to link `![img](path)` from source to the JSON output if the LLM forgets the explicit `image_id`.

### Q: Job stuck at "Rendering Manim"?
*   **Reason**: A Manim code generation failed or timed out.
*   **Fix**: Use the `/job/<id>/retry_phase` endpoint with `phase="manim_codegen"`.

## 6. ASCII Workflow: Manim Generation Details

```
┌────────────────────────────────────────────────────────┐
│               MANIM GENERATION PIPELINE                │
└────────────────────────────────────────────────────────┘

1. PARTITION DIRECTOR OUTPUT (JSON)
   └── "render_spec": { 
         "manim_scene_spec": "A circle grows...",
         "video_prompts": [...] 
       }

2. PIPELINE BRIDGE (pipeline_unified.py)
   ├── Filter: renderer == "manim"
   └── ThreadPool: Submit to ManimCodeGenerator

3. EXECUTOR (core/agents/manim_code_generator.py)
   ├── READ: manim_scene_spec
   ├── PROMPT: Load core/prompts/manim_system_prompt_v25.txt
   └── CALL: Claude Sonnet 4.5

4. VALIDATOR (Local)
   ├── CHECK: Syntax (compile check)
   ├── CHECK: Safety (no os.system)
   └── CHECK: V2.5 Rules (no Unicode, no absolute paths)

5. RENDERER (Local Docker/Process)
   ├── EXEC: manim -qm section_1.py GenScene
   └── OUT: /videos/section_1.mp4
```

## 7. February 2026 Updates

### 7.1 Blueprint Ready Flag

**Problem:** Users had to wait for all assets (videos/avatars) to complete before accessing the player, even though `presentation.json` was ready much earlier.

**Solution:** Added `blueprint_ready` field to job status.

**Implementation:**
- **Location:** `core/pipeline_unified.py` (line 476-485)
- **Trigger:** Set to `true` immediately after `presentation.json` is saved
- **API Response:** Automatically included in `GET /job/<job_id>/status`

**Example Response:**
```json
{
  "job_id": "abc123",
  "status": "processing",
  "blueprint_ready": true,  // ✅ NEW FIELD
  "current_step_name": "Blueprint Ready - Rendering Assets...",
  "progress": 60
}
```

**Benefits:**
- Zero breaking changes (existing `status` field unchanged)
- Frontend can opt-in to enable early player access
- Backward compatible with old dashboards

---

### 7.2 WAN Task Tracking

**Problem:** WAN video generation was a black box with no visibility into progress, failures, or errors.

**Solution:** Created `wan_status.json` file in job directory with real-time progress tracking.

**File Structure:**
```json
// jobs/<job_id>/wan_status.json
{
  "state": "processing",
  "total_beats": 10,
  "completed_beats": 7,
  "failed_beats": 1,
  "progress_percent": 70,
  "started_at": "2026-02-10T18:00:00",
  "updated_at": "2026-02-10T18:36:00",
  "details": {
    "pending": [],
    "in_progress": [],
    "completed": ["topic_1_beat_0", ...],
    "failed": ["topic_2_beat_3"],
    "errors": {"topic_2_beat_3": "Generation failed"}
  }
}
```

**New API Endpoint:**
```
GET /job/<job_id>/wan_status
```

**Implementation:**
- `render/wan/kie_batch_generator.py`: Added `_update_status()` method
- `core/renderer_executor.py`: Passes `wan_status.json` path to generator
- `api/app.py`: New polling endpoint (mirrors `avatar_status` pattern)

**Status States:**
- `processing` - Generation in progress
- `completed` - All beats succeeded
- `completed_with_errors` - Some beats failed
- `not_found` - WAN generation not started

### 7.3 WAN Task ID Persistence & Crash Recovery

**Problem:** If the server crashes or connection drops during WAN video polling, task IDs are lost. Videos may complete on Kie.ai but cannot be retrieved, requiring full regeneration.

**Solution:** Save Kie.ai task IDs to `wan_status.json` before polling, enabling recovery after crashes.

**Implementation:**
- **File:** `render/wan/kie_batch_generator.py`
- **New Methods:**
  - `_load_pending_tasks()`: Reads saved task_ids from status file
  - `resume_polling()`: Polls orphaned tasks from previous session
- **Modified Methods:**
  - `_update_status()`: New `save_pending_tasks=True` parameter to persist task_ids
  - `generate_batch()`: Auto-resumes orphaned tasks on startup (`resume_first=True`)

**Status File Structure:**
```json
// jobs/<job_id>/wan_status.json (during polling phase)
{
  "state": "polling",
  "pending_tasks": [
    {
      "beat_id": "topic_1_beat_0",
      "task_id": "kie_abc123",
      "output_path": "/path/to/video.mp4",
      "prompt": "A cinematic shot of...",
      "duration": 5
    }
  ],
  "total_beats": 10,
  "completed_beats": 3,
  ...
}
```

**Recovery Flow:**
1. Server submits tasks to Kie.ai → saves task_ids to `wan_status.json`
2. Server crashes during polling
3. On restart, `resume_polling()` reads saved task_ids
4. Polls Kie.ai for completed videos → downloads them
5. Only regenerates truly failed videos

**Benefits:**
- Zero video loss on server crashes
- No duplicate Kie.ai billing for already-submitted tasks
- Automatic recovery - no manual intervention needed

### 7.4 NSFW Prompt Persistence Fix

**Problem:** When WAN auto-corrects NSFW prompts using LLM:
- ✅ Video generates successfully (with sanitized prompt)
- ✅ Video paths saved to `presentation.json`
- ❌ Sanitized prompts NOT saved → Retry loops with original unsafe prompts

**Solution:** Modified `_update_presentation_safely` to write sanitized prompts back to `presentation.json`.

**Implementation:**
- **File:** `core/renderer_executor.py`
- **Line 450:** Pass `wan_beats` (with sanitized prompts) to `_update_presentation_safely`
- **Lines 538-556:** Update `video_prompts` array with sanitized versions
- **Thread Safety:** Uses `presentation_lock` for safe concurrent writes

**Flow:**
1. NSFW error detected → Auto-sanitize with LLM
2. Retry with sanitized prompt → Video generates
3. `wan_beats` updated in-memory with sanitized prompt
4. `_update_presentation_safely` matches `beat_id` in `video_prompts`
5. Writes sanitized prompt to `presentation.json` with lock
6. Future retries use safe prompt (no retry loop)

**V2.5 Retry Endpoint:**
- **Recommended:** `POST /api/v15/generate` with V2.5 parameters
- **Supports:** Automatic NSFW sanitization and prompt persistence
- **Deprecated:** Legacy `/jobs/<job_id>/rerender` (does not support auto-sanitization)

### 7.4 Granular Phase Retry

**Problem:** Users needed to retry specific parts of the pipeline (e.g., just one failed Manim video or a glitchy Avatar) without restarting the entire job and losing cost/progress.

**Solution:** Implemented `POST /job/<job_id>/retry_phase` for targeted regeneration.

**Capabilities:**
*   **Phases:**
    *   `manim_codegen`: Regenerates Manim code (and subsequently renders video).
    *   `manim_render`: Re-renders existing Manim code (useful for timeout/environment issues).
    *   `wan_render`: Re-submits WAN video generation. **Crucially, this detects and deletes "placeholder" videos** to force fresh generation.
    *   `avatar_generation`: Re-generates avatars, supporting multi-language.
*   **Granularity:** Supports `section_ids` list to target specific sections (e.g., `[3, 5]`).

**API Usage:**
```json
POST /job/<job_id>/retry_phase
{
  "phase": "wan_render",
  "section_ids": [4, 8] // Optional. If Omitted, retries all failed sections in phase.
}
```

### 7.7 Job ID Customization & IP Tracking

**Problem:** Standard UUIDs are hard to track. Users wanted to tag jobs with a project name, and admins needed to track job origins (IP addresses).

**Solution:** Updated `POST /submit_job` to accept a `job_prefix`.

**Capabilities:**
*   **Custom Prefix:** Users can optional send a `job_prefix` (e.g., "MyProject").
*   **IP Fallback:** If no prefix is provided, the system defaults to the requester's IP address (sanitized).
*   **Sanitization:** All prefixes are sanitized (alphanumeric + underscore only) to ensure filesystem safety.
*   **Format:** `[PREFIX]_[UUID]` (e.g., `MyProject_abc12345` or `192_168_1_50_abc12345`).

### 7.8 Job Analytics

**Problem:** Users lacked visibility into the cost and performance of their jobs (e.g., how many tokens were used, how long each phase took).

**Solution:** Implemented detailed tracking in `analytics.json` and a dedicated endpoint.

**Capabilities:**
*   **Token Tracking:** Logs input/output tokens for every LLM call (Plan, Content, Manim, etc.).
*   **Cost Estimation:** Calculates estimated cost based on model pricing (e.g., Gemini 1.5 Pro/Flash).
*   **Timing:** Tracks duration for each phase (LLM generation, rendering, etc.).

**API Usage:**
```
GET /job/<job_id>/analytics
```

---

### 7.6 Sanity Report API

**Problem:** Occasionally, assets (videos/images) might be missing despite the job being marked as "completed", usually due to file system issues or interrupted downloads.

**Solution:** A read-only endpoint that scans the actual disk assets and compares them against the `presentation.json` metadata.

**Capabilities:**
*   **Validation:** Checks existence of `presentation.json`, `source.md`, and all referenced video/image files.
*   **Granular Reporting:** Identifies exactly which section or beat is missing its asset.

**API Usage:**
```
GET /api/sanity-report/<job_id>
```
**Response:**
```json
{
  "job_id": "...",
  "status": "passed" | "failed",
  "missing_assets": [
    "videos/topic_1_beat_2.mp4",
    "avatars/en/section_3_avatar.mp4"
  ]
}
```

## 8. API Reference - V2.5 Endpoints
