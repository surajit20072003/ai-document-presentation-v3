## 1. Philosophy

| Principle | Rule |
|---|---|
| **V3 Core Goal** | Next-level visual learning — Three.js animations illustrate topics **in perfect sync with avatar narration**. When narrator says "look at this", the visual appears at that exact moment. |
| **Visual-first** | Students see before they read. Three.js replaces all text slides. |
| **Avatar = master clock** | `avatar.onended` is the only scene advance trigger. Player has zero timing logic. |
| **LLM owns sync** | Director writes narration duration AND animation duration in the same call. They always match exactly. |
| **Enforcer guarantees sync** | Phase 3.6 (`threejs_timing_enforcer.py`) patches `SEG_DUR[]` in .js files post-avatar, same as V2 Manim's `_enforce_timing()`. LLM timing is a starting point, not final truth. |
| **Zero text on screen** | `text_layer: "hide"` always. No exceptions. |
| **Mandatory quiz** | Every content/example section outputs an `understanding_quiz`. No skip condition. |

## 2. Renderer Decision Tree

```
1. Subject-based default:
   Math / Physics / Chemistry / Optics / Logic → threejs
   Biology / Anatomy / Human Organs            → video (WAN API, no local GPU)
   History / Geography / General               → video (local GPU)
   Intro / Summary / Memory / Recap            → none

2. Source image present (overrides step 1 for that beat):
   → analyse image → pick image_mode
   → image_mode: video_reference     → renderer: video
   → image_mode: texture             → renderer: threejs
   → image_mode: interactive_hotspot → renderer: threejs
```

---

## 3. Image Mode Decision Tree

| Condition | image_mode |
|---|---|
| Photograph / realistic illustration | `video_reference` |
| Labelled diagram — parts need individual explanation | `interactive_hotspot` |
| Diagram, chart, graph, table (no per-part clicking needed) | `texture` |

**Hard rules:**
- `image_mode` set → `image_source` must also be set (validator: hard fail)
- `video_reference` → `renderer: "video"`, no `threejs_file` (validator: hard fail)
- `texture` or `interactive_hotspot` → `renderer: "threejs"` (validator: hard fail)
- `interactive_hotspot` → `interaction` must be non-null (use `hover_highlight` or `click_reveal`)

## 4. Interaction Taxonomy

| Type | Student Action | Complexity Required |
|---|---|---|
| `hover_highlight` | Mouse over object → glows + tooltip | any |
| `click_reveal` | Click object → label/formula appears | any |
| `drag_point` | Drag vertex/point → geometry redraws | medium or complex |
| `rotate_inspect` | Drag canvas → 3D object rotates | medium or complex |
| `slider` | Drag handle → value changes | complex only |

**Decision rules:**
```
complexity: simple  → hover_highlight or click_reveal only
complexity: medium  → hover_highlight, click_reveal, drag_point, rotate_inspect
complexity: complex → any type; hover_highlight may be added as passive secondary

NEVER set interaction if:
  - segment_duration_seconds < 8.0
  - This is beat_0 of the section
  - Beat is purely narrative with no manipulable object
  - Two primary types would be needed

NEVER combine: drag_point + rotate_inspect
NEVER combine: slider + drag_point
```

**Interaction object schema (when non-null):**
```json
{
  "type":            "<one of 5 types>",
  "target":          "<name of interactive object>",
  "description":     "<one sentence — what student does + what changes>",
  "timeout_seconds": 10
}
```

## 5. Understanding Quiz (Mandatory, No Skip)

Every `section_type: content` or `section_type: example` section MUST output:

```json
"understanding_quiz": {
  "question": "Question text?",
  "options": { "A": "...", "B": "...", "C": "...", "D": "..." },
  "correct": "B",
  "explanation": "One sentence explaining why B is correct.",
  "narration": {
    "question_script": "Avatar reads question aloud including all 4 options.",
    "correct_script": "Avatar reacts positively and explains the answer.",
    "wrong_script": "Avatar corrects the student and explains the right answer.",
    "option_reveal_seconds": [3.5, 5.8, 8.1, 10.4]
  }
}
```

**Rules:**
- 4 options (A/B/C/D) — 3 options will fail in the validator
- `narration` sub-object with all 3 scripts is mandatory (used to generate 3 avatar clips)
- `narration.option_reveal_seconds` — **Required.** 4 floats, one per option. Seconds within the `question_script` clip when each option button should visually appear on screen, in sync with the avatar reading that option aloud.
  - Estimate using ~130 wpm: count words in the question intro before "Is it A:", divide by 130/60 → that gives you the timestamp for option A. Each subsequent option adds the word-count of the previous option text ÷ 130/60.
  - Example: 14-word intro (~6.5s) + 2-word options → `[6.5, 8.0, 9.5, 11.0]`
- No skip condition — even image-heavy sections get one question

## 7. `threejs_spec` Writing Guide (8-Item Checklist)

Every `threejs_spec` must cover, in order:

1. **Scene description** — what objects exist at t=0, their initial state
2. **Animation sequence** — what appears, in what order, at what second-level timing
3. **Interactivity** — which interaction type, what object, what changes; `none` if not interactive
4. **Camera** — position, angle, any movement
5. **Colours** — background (#0d1117), objects (use CLR palette names: gold, teal, rose, sky, lav, grn)
6. **Labels** — exact text that should appear, written as plain text (not LaTeX)
7. **Duration** — exact seconds matching `segment_duration_seconds`
8. **Complexity** — `simple | medium | complex`

Minimum 100 words. Shorter specs produce poor Three.js code.

## 9. Hard Rules (Non-Negotiable)

1. LLM owns sync — Director writes narration + animation duration together. Player never adjusts.
2. Avatar is master clock — `avatar.onended` is the only advance trigger.
3. No text in content — `text_layer: "hide"` always in content/example sections.
4. Three.js freeze on completion — never loop, never clear; freeze last frame.
5. 3 avatar clips per quiz question — question, correct, wrong. Always.
6. Understanding quiz is mandatory — every content/example section, no skip condition.
7. All `threejs_spec` 100+ words — shorter fails.
8. Gesture fallback is silent — camera denied = touch/click, no error shown.
9. Pinch zoom on Three.js only — WAN video and quiz overlay do not zoom.
10. Interactivity is opt-in — `interaction: null` is the default. Never invent it.
11. One interaction type per beat — no combining drag_point+rotate_inspect, slider+drag_point.
12. `timeout_seconds` mandatory — every interactive beat must have it. Player auto-advances when it fires.
13. Timeout ≠ duration extension — student interacts within the fixed window. Timeout is a safety net only.
14. **Never edit job-folder player copies** — `player/player_v3.html` is the one true master. `app.py` always serves it directly. Editing files under `player/jobs/` is wasted work.

## 11. V3 Sync Pipeline

**Updated:** 2026-03-09 — All 3 VSYNC issues resolved. See `V3_SYNC_ISSUES.md`.

The sync pipeline guarantees avatar narration and Three.js animation play in lockstep:

```
Phase 1: Director   → writes narration text + threejs_spec per section
Phase 3: Estimate   → word-count math sets duration_seconds per segment (fast, no audio)
Phase 3.5: Three.js → LLM generates .js using proportional SEG_DUR[] (VSYNC-002)
                       params.getTime() binds JS clock to av-vid.currentTime (VSYNC-003)
Phase 5: Avatar GPU → renders MP4 with voice baked in, writes avatar_duration_seconds (VSYNC-001A)
Phase 3.6: Enforcer → patches SEG_DUR[] in .js to match real MP4 duration (VSYNC-001B)
Player:             → _avBox guard: animation holds at t=0 until vid.onplay fires (VSYNC-003)
                       params.getTime() → av-vid.currentTime = avatar MP4 is the only clock
```

### Key Files

| File | Role |
|---|---|
| `core/threejs_timing_enforcer.py` | Phase 3.6 — patches `SEG_DUR[]` arrays post-avatar |
| `core/prompts/threejs_system_prompt_v2.txt` | PROPORTIONAL SYNC pattern for LLM |
| `core/prompts/threejs_user_prompt.txt` | Narration timeline + proportional sync rule |
| `player/player_v3.html` | `_avBox` cold-start guard + `params.getTime()` avatar clock |
| `core/agents/avatar_generator.py` | Writes `avatar_duration_seconds` after MP4 download |

### V2 → V3 Sync Comparison

| Feature | V2 (Manim) | V3 (Three.js) |
|---|---|---|
| Duration source | TTS MP3 audio file | Word-count estimate → corrected by Phase 3.6 |
| Sync enforcer | `_enforce_timing()` patches `self.wait()` | Phase 3.6 patches `SEG_DUR[]` |
| Audio playback | TTS MP3 played by player | Avatar MP4 only — voice baked in |
| Clock source | Manim renders at exact TTS timing | `params.getTime()` → `av-vid.currentTime` |
| Cold-start drift | None (pre-rendered) | Fixed via `_avBox` guard |
| Result | ✅ Perfect sync | ✅ Perfect sync |

## 12. Quiz Visual Layer

**Updated:** 2026-03-09 — QUIZ-001 resolved. See `V3_SYNC_ISSUES.md`.

### Architecture

Every quiz question has TWO outputs from the pipeline:

| Output | Generated by | Purpose |
|---|---|---|
| `threejs/quiz_card_{sec_id}.js` | `quiz_card_generator.py` Phase 4.5 (no LLM) | Visual layer — A/B/C/D option card displayed while avatar reads question |
| `avatar_quiz_{id}_q{n}_{question\|correct\|wrong}.mp4` | `avatar_generator.submit_quiz_clips()` Phase 6 | Voice — avatar reads question, correct feedback, wrong feedback |

### Quiz Flow (Player)

```
1. Player reads section_type == 'quiz'
2. Loads quiz_threejs_file .js → initScene() → A/B/C/D card visible (all hidden)
3. Plays avatar quiz_{id}_q{n}_question.mp4 (avatar reads question aloud)
4. A/B/C/D buttons reveal one-by-one in sync with av-vid.currentTime:
   - Player reads narration.option_reveal_seconds [A_t, B_t, C_t, D_t]
   - Each button slides in when av-vid.currentTime >= that timestamp
   - Fallback: even-split across clip duration if option_reveal_seconds absent
5. Student clicks → params.onInteract('click', {key, correct}) fires
   → IF correct: correct tile glows green → play _correct.mp4
   → IF wrong:   chosen tile glows red, correct glows green → play _wrong.mp4
6. Avatar clip ends → advance to next question or next section
```

### Quiz Card `.js` Contract

```javascript
// Programmatic — generated by quiz_card_generator.py
// Implements the same initScene() contract as Three.js content scenes
initScene(container, totalDuration, params)
// Returns: { onResize, onPinchZoom, dispose, onPointerMove, onPointerDown, onPointerUp }
// params.onInteract('click', { key: 'B', correct: true }) fired on selection
```

### Director Rules for Quiz

| Rule | Requirement |
|---|---|
| `understanding_quiz` | **Mandatory** on every content/example section. No exceptions. |
| Options | Always 4 (A/B/C/D). Hard fail if < 4. |
| `correct` key | Must match one of A/B/C/D exactly. Hard fail if missing. |
| `narration.question_script` | Required — avatar reads this for the question clip. |
| `narration.correct_script` | Required — avatar reads this for the correct-answer clip. |
| `narration.wrong_script` | Required — avatar reads this for the wrong-answer clip. |
| `narration.option_reveal_seconds` | **Required** — 4 floats. Second within question_script clip when each option (A/B/C/D) visually appears, synced to avatar mouth. Estimate at ~130 wpm. Player uses this to drive `timeupdate` reveal. |
| `quiz_threejs_file` | Written by Phase 4.5 automatically — Director does not set this. |

### Avatar Overlay (WebGL Chroma Key)

| Setting | Value |
|---|---|
| Element | `#av-overlay` — fixed bottom-right corner |
| Default size | `160 × 284px` (9:16) |

## 14. Player Deployment Architecture

**Updated:** 2026-03-10 — INFRA-001 resolved.

### How `player_v3.html` is served

```
Flask route: /player/jobs/<job_id>/player_v3.html
  → app.py intercepts filename == "player_v3.html"
  → always send_from_directory(PLAYER_DIR, "player_v3.html")  ← master
  → never reads the per-job copy
```

**One file, one truth.** Any edit to `player/player_v3.html` is live on every job on the next browser refresh — no deployment, no copy, no restart needed.

### Why job folders exist with a `player_v3.html` copy

They are legacy artefacts from the original copy-on-create approach. The files are kept in place so the folder structure stays valid, but they are **never read** by the live server.

### What `setup_job_output_dir` does now

When a new job folder is created, `player_v3.html` is still copied once as a local backup. The copy is always overwritten (the old `not dst.exists()` guard is removed) so the backup stays current.

### Docker container

| Field | Value |
|---|---|
| Container name | `ai-document-presentation-v3-api-1` |
| Port mapping | `0.0.0.0:5006 → 5000` |
| Restart command | `sudo docker restart ai-document-presentation-v3-api-1` |
| Verify live | `curl -s -o /dev/null -w "%{http_code}" http://localhost:5006/health` |

### INFRA-001: Stale player copies bug (RESOLVED 2026-03-10)

**Problem:** `app.py` only copied `player_v3.html` to each job folder once (guarded by `not dst.exists()`). All subsequent edits to the master were invisible — each job served its own frozen snapshot.

**Impact:** All fixes deployed to `player_v3.html` (AVTR-002, AVTR-003, AVTR-004, QUIZ-002, THREEJS-001) were only visible in the one job folder that was manually re-deployed to. All other 35 jobs showed the old player.

**Fix:** Route intercept in `serve_job_assets()` (`api/app.py` line ~4586). `player_v3.html` requests short-circuit to `PLAYER_DIR` before the job-folder file check.
